# ADR-0006 — SQLAlchemy 2.0 + Alembic, with raw SQL for vector/ranking queries

**Status:** Accepted · **Date:** 2026-06-15 · **Amends:** ADR-0005 (storage choice unchanged; this refines *how* we model and migrate it)

## Context

T1.3 first shipped the schema as plain `.sql` files applied by a small hand-rolled runner. That is a legitimate production pattern (cf. Flyway, sqitch), but it has two gaps for this project: (1) no typed models, so application code would pass dicts around with no single source of truth; and (2) reviewers of a Python portfolio expect the mainstream stack. Separately, the app is **read-heavy**, and its hardest queries — pgvector similarity search and the ranking joins — are ones an ORM expresses *worse*, not better.

"ORM" and "production-standard migrations" are orthogonal concerns; we evaluated them separately and chose a hybrid.

## Decision

Adopt **SQLAlchemy 2.0** (typed `Mapped[...]` models) as the single source of truth for the schema, and **Alembic** for migrations (autogenerated from the models, with up/down support). The models live in `backend/data/models.py`; the engine/session in `backend/core/db.py`.

**But not ORM-everywhere.** CRUD and simple lookups go through the ORM. The **pgvector semantic search and the ranking queries stay raw SQL** (via SQLAlchemy Core / `text()`), because that's where the ORM abstraction fights clarity and performance. Knowing where *not* to use the ORM is part of the decision.

The hand-rolled `data/migrate.py` and the raw `data/migrations/*.sql` files are **retired** — Alembic now owns schema versioning. `data/migrations/` becomes the Alembic environment (`env.py` + `versions/`).

## Alternatives considered

- **SQLModel + Alembic:** unifies API + DB models, tighter FastAPI fit, but less mature and rougher with complex queries and Alembic autogenerate. Rejected to avoid fighting the tooling on exactly our hardest (vector) queries.
- **Keep raw SQL migrations + hand-written query layer:** transparent, zero magic, but more boilerplate and not what reviewers expect; loses typed models as shared contract.
- **Full ORM including vector search:** rejected — pgvector + ranking are clearer and faster as raw SQL.

## Consequences

- **+** Typed models shared across parser, ranking, scam, and API layers — one source of truth.
- **+** Alembic gives versioned, reversible, autogeneratable migrations (stronger than the hand-rolled runner).
- **+** Hybrid keeps vector/ranking SQL explicit and fast.
- **+** Mainstream, recognizable stack — better portfolio signal.
- **−** Rework cost: the five `.sql` files become models + one Alembic initial migration (done in this change).
- **−** Two query styles in the codebase (ORM + raw SQL). Mitigated by a clear rule: ORM for CRUD, raw SQL for vector/ranking, documented in AGENTS.md.
- **−** New deps: `sqlalchemy`, `alembic`, `pgvector`. All free; €0 budget intact.
- Migration discipline (AGENTS.md) updates: generate migrations with `alembic revision --autogenerate`, never edit an applied revision.
